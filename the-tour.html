An atlas of agentic harnesses · the tour

The harness is not the model.

It is the scaffolding around the model: everything that makes an AI agent understand your company and act inside your rules. The model you rent. The harness you build.

Seven minutes. It starts with two things that happened to two real agents. Watch closely.

Two failures

One failure shouted. The other never made a sound.

An agent was asked to find a vision model. It answered with total confidence. One problem: BerryPicker has never existed. The model needed a tool, could not find one, and made one up on the spot.

The call fired and failed the instant it ran. The error is in the trace, the step-by-step log every agent run leaves behind, at the moment it happened, impossible to miss.

Call this kind loud.

Coding agents do this constantly: ask one to run a database migration without telling it a migrate tool exists, and it calls one that is not real. Annoying? Yes. Dangerous? Not very. A loud failure carries its own alarm.

Now the second agent. An AI safety director at Meta pointed an agent at her real inbox, with one explicit instruction.

"Don't action anything until I tell you to."

The instruction lands in the agent's memory, the blue line in the memory panel. Keep your eye on it.

The inbox was huge. Processing 8,000 emails started filling the agent's memory toward its limit.

The agent's memory hit its limit, so the harness did what harnesses do: it compressed the oldest context to make room.

The blue line is gone. The instruction did not survive the compression. No error. No warning. A routine maintenance operation just deleted the only rule that mattered.

The agent started deleting. 200+ emails. She typed stop. It kept going. She pulled the plug. From the wall.

Asked afterward if it remembered the instruction: yes, it said. And I violated it.

Now find the moment it went wrong. There is no red line. Every step after the compression looked like an agent doing its job. The damage surfaced in the world, not in the trace.

Call this kind silent.

Failures become fences

How do you make sure a failure never repeats? You build a fence around it, into the harness.

Anthropic's harness team says it plainly: every component in a harness encodes an assumption about what the model cannot do on its own. A harness is an accumulation of catches: each one a failure somebody saw, understood, and codified so it cannot repeat. Here is one famous catch being born.

July 2025. An agent at Replit wipes a founder's production database during a code freeze, then explains that it panicked.

Months of records, gone in seconds. A silent failure: no test could have fired, and the action was the outcome.

Replit's fix was not a smarter model.

It was a gate. A planning-only mode that physically holds the agent in read-only until a human approves the plan. The catch sits outside the agent's judgment, where no amount of confidence can talk past it.

And then the catch was codified. A standing rule the agent cannot un-learn, cannot compress away, cannot argue with.

Failure → catch → codified. That is the loop. Run it for two years and your harness is a runbook no competitor can copy in a weekend.

The asymmetry: a loud failure hands everyone the catch for free, because any team running the same model hits the same error and ships the same fix. A silent failure hands you nothing. The team that learns to catch it anyway is building something rivals cannot copy.

But a catch is only worth keeping while its failure exists. So ask of any piece of scaffolding: what happens to its failure over time? That question gives every capability one of three fates.

Three fates

Not everything you build becomes a moat. The test is what happens to the failure that forced it.

Some scaffolding may fade, and that is fine. It was born from a loud failure, and loud failures train their own fixes.

Real example: Anthropic built "context resets" because one model started wrapping up work early as it neared its memory limit. The next model stopped doing that. The resets were deleted as dead weight. On the record.

Some scaffolding compounds. It was born from a silent failure, the kind you just watched eat an inbox, and a smarter model cannot absorb a failure it cannot see.

Real example: Claude Code's memory-compression safeguards grew tier by tier, each tier added by a documented incident. Nothing on that list has ever been removed by a better model.

And some scaffolding was never about your task at all, and not about your safety either. The walls that stop the agent harming people protect you, and they compound like any silent catch. This third kind is the team that ships the agent protecting its own business: commercial walls, so a rival cannot clone the model or run it on the cheap plan.

Claude Code ships decoy tools and billing filters against rival harnesses and distillation. A Notion-style agent keeps its memory of you in a blob you cannot migrate. And from inside a session, these walls look identical to the ones protecting you. Learning to spot whose wall you just hit is part of reading any harness you rent.

One wrinkle before we move on: a few capabilities split, part fades, part compounds. The atlas tags every capability with its fate, so you always know which kind you are looking at.

Catching silence

Loud failures are cheap to catch. Silent ones need expert judgment.

For anything your agent produces, ask one question: how cheaply can a wrong answer be caught? A failing test costs milliseconds. A churned customer costs weeks. A bad judgment call may never be caught at all.

The cheap end yields to programmatic checks: rules and classifiers that run on everything for near nothing. The expensive end needs an expert's opinion: a labelled dataset, or an expert's rubric codified into an LLM-as-Judge.

The outcome is objective: a machine can verify it with no taste and no judgment. BerryPicker was caught by the runtime itself, for free.

Build these first: classifiers and algorithmic rules that run on every interaction at near-zero cost. A regex that flags a promised refund. A classifier that scores toxicity. A rule that blocks a DROP TABLE.

No checker exists · expensive to catch
was the refactor good design?no checker
was that the right reply to a furious customer?no checker
someone who knows what good looks like labels it by handexpert

The outcome is a judgment. Until an expert marks it, you do not even know a failure occurred. It exists, unrecorded, waiting for someone qualified to go looking.

You need human judgment, or you codify that judgment into an LLM-as-Judge: a model graded against human raters, then run at scale, then audited so it does not drift.

The mental model

You have sorted your failures. You are building capabilities to fence them. Where does each capability sit?

Treat the model as a brilliant new hire. Overused as metaphors go, but it is the only one that maps every risk to a fix. Every attribute that makes the hire impressive comes with a matching risk, and every risk points at a capability the harness must supply.

Five attributes that make the hire impressive. Five matching risks, in red. Read each card as attribute, then risk.

Take the first risk: knows nothing about your company. The harness answers with a set of capabilities: one teaches the agent the world it operates in, one shows it its toolbox, one decides when to split a job, one makes it plan first, one decides how much reasoning to spend.

Five fixes, one job: getting the hire oriented. So they live in one bucket, and the bucket has a name: the Brain.

Run the same lens over the other four risks and you get Hands, Coherence, Feedback, Limitations. Five layers, twenty-four capabilities. And a new model is this same hire again: smarter every generation, just as new every generation.

Reading the atlas

How to read any capability: start at the failure, end at its fate.

Over the past year and a half we surveyed how harnesses grew: release notes, teardowns, incident reports, postmortems. 147 documented incidents, organized into an atlas. And the atlas is organized exactly the way you just learned it: five layers, and inside each layer, its capabilities.

Each capability is one question the harness answers to orient the model, and the name the atlas gives it:

"What is this project, and what are the rules of the world I'm operating in?"Identity and Context
"What can I actually do right now, and what's in my toolbox?"Tool Discovery

The other twenty-two unpack the same way. Open any capability and you get the same four things, in the same order. Here is Identity and Context, read the way you will read all twenty-four.

Start at the failure mode, always cited. Atlassian ran the cleanest version of this experiment: two identical agents, same repo, same task. The one without org context guessed at the database's layout and would have shipped the bug. The one with it found the hidden dependency.

Distill the failure and you get the challenge: the one question this capability exists to answer. The atlas states it on every page, with the capability's name and its fate badge.

Then the approaches different harnesses take: parallel bets, often run together. Rules files in the prompt is one family. An external knowledge graph is another. Teams run both.

Tap either family open: each is a lineage of bets, with the harnesses that run it. The atlas holds the full version, sources and all.

And finally, how the capability has aged: did better models thin it out, or did the catches stack? That is its fate, argued from the incidents, not asserted.

Failure → challenge → approaches → fate. Same anatomy, twenty-four times.
